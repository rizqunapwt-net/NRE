# 🤝 Agent Handoff Protocol — NRE Multi-Agent Framework

> **Handoff Protocol** untuk transfer pekerjaan antar agent dengan smooth, clear, dan traceable.

## Overview

**Purpose**: Memastikan ketika satu agent selesai task → agent lain bisa langsung mulai dengan informasi lengkap, tanpa back-and-forth, tanpa missing context.

**When Handoff Happens**:
1. Agent selesai satu fitur/phase
2. Fitur itu **diperlukan** oleh agent lain
3. Handoff must be **explicit & documented**

## Handoff Scenarios

### Scenario A: Backend (A1) → Frontend (A2)

**When**: A1 finishes API endpoint → A2 consumes di frontend

**Checklist (A1 - Handoff Owner)**:
- ✅ API endpoint implemented & tested locally
- ✅ Request/response spec documented (method, URL, auth, body, response)
- ✅ Error handling implemented (400, 403, 404, 500)
- ✅ Seeder/dummy data available for A2 to consume
- ✅ Database schema stable (no more changes)

**Communication (A1 → A2)**:
```markdown
### [DATE TIME] Agent 1 → Agent 2
**API READY FOR CONSUMPTION**:
- Endpoint: GET /api/v1/public/catalog?category={id}&page={n}
- Auth: Public (no token required)
- Response:
  {
    "books": [
      {
        "id": 1,
        "title": "Book Title",
        "slug": "book-slug",
        "cover_url": "https://...",
        "price": "99000.00",
        "author": { "id": 1, "name": "Author Name" }
      }
    ],
    "pagination": { "total": 100, "per_page": 10, "current_page": 1 }
  }
- Status Code: 200 (success), 400 (bad query), 404 (not found)
- **Ready since**: [date]
- **Dummy data**: Yes, generated by seeder
- **Testing**: Tested with curl/Postman, working
```

**Checklist (A2 - Consumer)**:
- ✅ Received handoff notification
- ✅ Tested API endpoint locally
- ✅ Implements frontend UI consuming the API
- ✅ Reports back if API doesn't match spec
- ✅ Updates Communication Log when A2 is done with frontend component

**If A2 Finds Issue**:
```markdown
### [DATE TIME] Agent 2 → Agent 1
**API SPEC MISMATCH**:
- Issue: Response doesn't include `author.email`
- Expected: { id, name, email }
- Got: { id, name }
- Impact: Frontend needs to display author email on detail page
- Fix ETA: [when A1 can fix]
```

**Resolution** (A1):
```markdown
### [DATE TIME] Agent 1 → Agent 2
**ISSUE RESOLVED**:
- Fixed: Added `author.email` to response
- Tested: curl GET /api/v1/public/catalog/1 ✅
- Deployed: http://localhost:9000
```

### Scenario B: Backend (A1) → Admin Panel (A3)

**Same as Scenario A**, but for admin endpoints.

**Checklist (A1)**:
- ✅ `/api/v1/admin/books` CRUD endpoint done
- ✅ Role-based auth (admin only)
- ✅ Request/response spec documented
- ✅ Pagination, search, filtering implemented
- ✅ File upload (cover, PDF) working

**Communication**:
```markdown
### [DATE] Agent 1 → Agent 3
**ADMIN ENDPOINTS READY**:
- CRUD: POST /api/v1/admin/books, GET /api/v1/admin/books/{id}, PATCH, DELETE
- Auth: Bearer token + admin role
- File upload: multipart/form-data for cover_url, pdf_path
- Pagination: ?page=1&per_page=10
- Search: ?search=title
- Response: { id, title, slug, price, cover_url, pdf_path, ... }
```

**When**: A1 creates BookStorageService → A5 uses it to serve files

**Checklist (A1)**:
- ✅ `BookStorageService` methods documented
- ✅ Cover serving method ready
- ✅ PDF streaming method ready
- ✅ Thumbnail generation service ready
- ✅ Storage paths defined

**Communication**:
```markdown
### [DATE] Agent 1 → Agent 5
**STORAGE SERVICE READY**:
- Class: App\Services\BookStorageService
- Methods:
  - getCoverPath($book_id) → returns file path
  - getPdfPath($book_id) → returns file path
  - generateThumbnail($book_id, $size) → creates thumbnail
  - getDownloadUrl($book_id, $type) → returns download URL
- Storage disks: 'private' (uploaded), 'public' (imported)
- Example: BookStorageService::getCoverPath(1) → storage/app/private/...
```

```markdown
### [TIMESTAMP] Agent X → Agent Y
**HANDOFF: [Feature/Component Name]**

**What's Done**:
```

## Handoff Checklist (Universal)

**Before Any Handoff** ✅:
- [ ] Feature implemented & working locally
- [ ] Code committed to `develop` branch
- [ ] Tests written & passing
- [ ] Documentation updated (if applicable)
- [ ] No breaking changes to existing API/code
- [ ] Dependent agent notified via Communication Log
- [ ] Spec/contract clearly defined
- [ ] Code ready for code review
- [ ] MCP_STATE.md updated with task status

**Receiving Handoff** ✅:
- [ ] Read Communication Log message fully
- [ ] Understand spec/contract completely
- [ ] Test/verify endpoint/service locally
- [ ] Ask clarifying questions if needed (reply in Comms Log)
- [ ] Plan implementation based on spec
- [ ] Update own task to "IN_PROGRESS"
- [ ] Implement feature/component

**After Your Work** ✅:
- [ ] Test integration with handed-off code
- [ ] Commit code with proper message format
- [ ] Test with other agents' code if applicable
- [ ] Update MCP_STATE.md with task status
- [ ] Report completion in Communication Log
- [ ] Mark as ready for next agent (if applicable)

## Conflict Resolution During Handoff

**Conflict**: Agent Y disagrees with Agent X's spec/implementation

**Process**:
1. **Agent Y** posts concern in Communication Log with details
2. **Agent X** (or domain owner) responds with explanation
3. **Compromise** reached via discussion
4. **Decision** documented in MCP_STATE.md gotchas
5. **Both** update their code if necessary
6. **Test** integration again

**Example**:
```markdown
### [DATE] Agent 2 → Agent 1
**CONCERN: API Response Format**
- Issue: Expected pagination.total_count, got pagination.total
- Breaking change for frontend
- Request: Rename to total_count or add both fields

### [DATE] Agent 1 → Agent 2
**RESPONSE**:
- Reasoning: PostgreSQL constraint, total is standard naming
- Compromise: Keep total, add total_count alias in API transformer
- Fixed: Already updated API response
- Tested: curl endpoint verifies both fields present
```

## Tracking Handoffs

**All Handoffs are Logged In**:
1. `.agents/MCP_STATE.md` — Communication Log (main)
2. `.agents/ALERTS.log` — Server-side logging (if MCP server running)
3. Git commits — track code actually transferred

**Query Handoffs**:
```bash
# Via MCP Server
curl http://localhost:8888/api/comms?limit=20

# Or search MCP_STATE.md
grep "Agent X → Agent Y" .agents/MCP_STATE.md
```

## Best Practices

### ✅ DO

- **Be explicit** about what's ready & what's not
- **Document spec clearly** (request/response, auth, errors)
- **Test before handing off** — don't pass broken code
- **Be available** for questions during handoff period
- **Update progressively** — don't wait until everything is perfect
- **Reference commits** for traceability
- **Use consistent format** — follow template above

### ❌ DON'T

- **Hand off incomplete work** without clear status
- **Leave spec ambiguous** — use examples
- **Ignore receiver's questions** — respond quickly
- **Make breaking changes** without notification
- **Assume receiver understands** — over-communicate
- **Skip testing** before handoff
- **Forget to update MCP_STATE.md** & Communication Log

---

## Success Criteria for Handoff

✅ Handoff is **successful** if:
- Receiver understands spec on first read
- Receiver can run/test locally immediately
- No back-and-forth clarifications needed
- Receiver can start work within 1 hour
- Code integrates smoothly when receiver finishes
- No surprises or missing dependencies
- MCP_STATE.md task tracking stays updated
